#  АІС «ZLAGODA» - Керування процесом збуту товарів

Автоматизована інформаційна система для регулювання роботи продуктового міні-супермаркету **«ZLAGODA»**. Система автоматизує процеси обробки даних, що виникають під час збуту товарів: облік товарів, категорій, чеків, персоналу та постійних клієнтів.

##  Мета розробки

Забезпечення зручного доступу до інформації для касирів та менеджерів міні-супермаркету. Система дозволяє відстежувати кількість та ціну продажу одиниць усіх товарів, наявних у магазині, історію збуту, список типів товарів та їхні характеристики, інформацію про всіх працівників та постійних покупців.

---

## Ролі користувачів

Система розрахована на два типи користувачів: **Менеджер** та **Касир**.

### Менеджер
Відповідає за контроль усіх процесів у міні-супермаркеті:
- Додавання, редагування та видалення даних про працівників, постійних клієнтів, категорії товарів, товари, товари у магазині та чеки
- Друк звітів з інформацією про всі сутності системи
- Перегляд аналітики продажів: за касирами, товарами та часовими періодами

### Касир
Відповідає за здійснення продажу товарів:
- Створення чеків та здійснення продажу (**виключне право касира** — менеджер не має доступу до цієї операції)
- Пошук товарів за назвою, категорією або UPC
- Додавання та редагування інформації про постійних клієнтів
- Перегляд власних чеків за день або обраний період

---

##  Функціональні вимоги

### Менеджер
1. Додавати/редагувати/видаляти дані про працівників, клієнтів, категорії, товари, товари у магазині, чеки
2. Друкувати звіти по всіх сутностях системи (працівники, клієнти, категорії, товари, товари у магазині, чеки)
3. Отримувати перелік всіх працівників, відсортованих за прізвищем
4. Отримувати перелік касирів, відсортованих за прізвищем
5. Знаходити телефон та адресу працівника за його прізвищем
6. Отримувати перелік постійних клієнтів з певним відсотком знижки, відсортованих за прізвищем
7. Здійснювати пошук товарів за категорією, відсортованих за назвою
8. Шукати інформацію про товар (ціна, кількість, назва, характеристики) за UPC
9. Отримувати перелік акційних та не акційних товарів, відсортованих за кількістю або назвою
10. Переглядати чеки, створені конкретним касиром або всіма касирами за обраний період (з деталізацією по товарах)
11. Визначати загальну суму продажів за касиром або всіма касирами за обраний період
12. Визначати загальну кількість одиниць певного товару, проданого за певний період

### Касир
1. Отримувати перелік всіх товарів та товарів у магазині, відсортованих за назвою
2. Здійснювати пошук товарів за назвою або категорією
3. Здійснювати пошук постійних клієнтів за прізвищем
4. Знаходити ціну та кількість товару за UPC
5. Здійснювати продаж товарів (додавання чеків) — **виключне право, недоступне менеджеру**
6. Додавати та редагувати інформацію про постійних клієнтів
7. Переглядати власні чеки за поточний день або обраний період
8. Отримувати повну інформацію про конкретний чек за його номером
9. Отримувати інформацію про себе
10. Отримувати перелік акційних товарів, відсортованих за кількістю одиниць або назвою
11. Отримувати перелік не акційних товарів, відсортованих за кількістю одиниць або назвою

##  Бізнес-логіка

### ПДВ
ПДВ складає **20%** від уже сформованої ціни продажу товару та **входить** у загальну суму чека. Розраховується як: `загальна_сума * 0.20`.

### Акційні товари
Ціна акційного товару розраховується від ціни звичайного шляхом застосування стандартної знижки **20%**:
```
акційна_ціна = ціна_товару * 0.8
```
Товар стає акційним, якщо термін придатності скоро збігає, а залишок на складі є значним.

### Переоцінка товарів
При надходженні нової партії товару за новою ціною продажу відбувається **переоцінка всього залишку**: усі одиниці цього товару (і старі, і нові) отримують нову ціну продажу.

### Знижка за картою клієнта
Сума чека може бути знижена за допомогою однієї карти клієнта з визначеним відсотком знижки.

### Зберігання чеків
Інформація про всі чеки зберігається у базі даних протягом **3 років**. Чеки, старші за 3 роки, підлягають автоматичному видаленню (реалізується через scheduled job або pg_cron).

## Модель даних

Система базується на реляційній моделі, що складається з **6 основних таблиць** та таблиці зв'язку:

| Таблиця | Опис | Ключові поля |
|---|---|---|
| `Employee` | Працівники супермаркету | `id_employee` (PK), ПІБ (прізвище, ім'я, по батькові\*), посада, зарплата, дата початку, дата народження, телефон, адреса (місто, вулиця, індекс) |
| `Category` | Категорії товарів | `category_number` (PK), назва |
| `Product` | Загальна інформація про товари | `id_product` (PK), назва, виробник, характеристики |
| `Store_Product` | Товари у магазині (UPC, ціна, залишок) | `UPC` (PK), `UPC_prom` (FK, самопосилання — nullable), `id_product` (FK), ціна продажу, кількість (`products_number`), акційний (bool) |
| `Customer_Card` | Карти постійних клієнтів | `card_number` (PK), ПІБ (прізвище, ім'я, по батькові\*), телефон, адреса (місто\*, вулиця\*, індекс\*), відсоток знижки |
| `Check` | Заголовки чеків | `check_number` (PK), `id_employee` (FK), `card_number` (FK), дата, загальна сума, ПДВ |
| `Sale` | Товари у чеку (зв'язок M:M) | `UPC` (PPK, FK1), `check_number` (PPK, FK2), `products_number` (кількість), ціна на момент продажу |

> \* — необов'язкове поле (nullable)

> **Важливо:** Ціна та кількість товару фіксуються у таблиці `Sale` на момент продажу, оскільки після переоцінки вони можуть змінитись.

> **Важливо:** Поле `UPC_prom` у `Store_Product` — це самопосилання (FK на `UPC` тієї ж таблиці), що зв'язує звичайний товар з його акційним варіантом. Один товар (`Product`) може мати не більше **двох** записів у `Store_Product` (звичайний + акційний).

##  Архітектура проєкту

Проєкт побудовано за **багаторівневою архітектурою** з чітким розподілом відповідальності між шарами.

```

              Frontend (React)               
   Інтерфейс Менеджера / Касира + Звіти     
                    HTTP (REST API)
                   
            FastAPI (Backend)               
   Routers -> Services -> Repositories (DAO)  
                  Pure SQL (psycopg2)
                  
         PostgreSQL (Database)             
  Employee, Product, Check, Sale, ...        
```

### Шари застосунку

**1. Рівень маршрутизації - `routers/`**
FastAPI Router розбиває API на логічні модулі (`/employees`, `/products`, `/checks` тощо). Кожен роутер перевіряє права доступу через залежності (`Depends`). Ендпоінт `POST /checks` доступний **виключно касиру**.

**2. Рівень сервісів - `services/`**
Містить бізнес-логіку:
- Розрахунок акційної ціни (`price * 0.8`)
- Обчислення ПДВ (`sum * 0.20`)
- Переоцінка всіх залишків при надходженні нової партії
- Валідація вікового обмеження для працівників (18+)

**3. Рівень доступу до даних - `repositories/`**
Кожна сутність має свій репозиторій з чистими SQL-запитами через `psycopg2`. ORM **не використовується**.

**4. Схеми даних - `schemas/`**
Pydantic-моделі для валідації вхідних та вихідних даних API.

##  Технологічний стек

| Інструмент | Призначення | Які вимоги закриває |
|---|---|---|
| **FastAPI** | Основний веб-фреймворк, REST API | Зручний інтерфейс для фронтенду, автодокументація (Swagger UI), валідація через Pydantic |
| **PostgreSQL** | Реляційна база даних | Зберігання 6 таблиць із зовнішніми ключами, каскадні дії, обмеження цілісності (`CHECK`, `NOT NULL`) |
| **psycopg2** | Драйвер підключення до PostgreSQL | Виконання **чистих SQL-запитів** без ORM — ключова вимога системи |
| **Pydantic v2** | Валідація та серіалізація даних | Перевірка обмежень (від'ємні значення, довжина телефону), типізація схем запитів та відповідей |
| **python-jose / PyJWT** | Генерація та перевірка JWT-токенів | Реалізація **аутентифікації** на основі токенів |
| **Passlib + bcrypt** | Хешування паролів | **Шифрування паролів** працівників при зберіганні у БД |
| **FastAPI Security (OAuth2)** | Механізм авторизації | Розмежування прав доступу між роллю **Менеджер** та **Касир** (RBAC), включаючи виключне право касира на створення чеків |
| **ReportLab / WeasyPrint** | Генерація PDF-звітів | **Друк звітів** по всіх 6 сутностях системи |
| **React** | SPA-фронтенд | **Зручний та інтуїтивно зрозумілий UI** для обох ролей |
| **Alembic** | Міграції бази даних | Версіонування схеми БД, відтворюване розгортання |
| **Docker / Docker Compose** | Контейнеризація | Ізольоване середовище для FastAPI + PostgreSQL |

##  Безпека

- **Аутентифікація** - при вході перевіряється логін та пароль працівника; у відповідь повертається JWT-токен.
- **Авторизація (RBAC)** - кожен ендпоінт через `Depends` перевіряє роль із токена. Менеджер має повний CRUD-доступ (крім створення чеків); Касир має виключне право на створення чеків (`POST /checks`) та обмежені операції з клієнтами.
- **Шифрування паролів** - паролі зберігаються у БД виключно у вигляді bcrypt-хешу.
- **Заборона ORM** - вся взаємодія з БД відбувається через чисті SQL-запити (`psycopg2`), що забезпечує повний контроль над виконуваними запитами.

## Обмеження цілісності

Наступні бізнес-правила реалізовані **одночасно** на рівні бази даних (`CHECK`-обмеження у PostgreSQL) та на рівні Pydantic-схем:

- Вік працівника - не менше **18 років**:
  ```sql
  CHECK (CURRENT_DATE - date_of_birth >= INTERVAL '18 years')
  ```
- Номер телефону - не більше **13 символів** (включаючи `+`):
  ```sql
  CHECK (length(phone_number) <= 13)
  ```
- Ціна, кількість, зарплата, відсоток знижки - **не можуть бути від'ємними**:
  ```sql
  CHECK (value >= 0)
  ```
- Один товар (`Product`) може мати не більше **двох** записів у `Store_Product` (звичайний + акційний)
- Поле `empl_patronymic` у `Employee` та `cust_patronymic` у `Customer_Card` є **nullable** (по батькові може бути відсутнє)
- Адреса зберігається як **три окремі поля**: `city`, `street`, `zip_code` (для `Employee` - обов'язкові; для `Customer_Card` — nullable)
